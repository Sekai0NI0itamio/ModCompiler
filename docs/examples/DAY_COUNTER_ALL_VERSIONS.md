# Day Counter (Optimized Day Counter) — All-Versions Port

## Overview

**Mod**: Optimized Day Counter (https://modrinth.com/mod/optimized-day-counter)  
**Modrinth slug**: `time-counter`  
**What it does**: Client-side HUD mod that shows the current world day and optional clock  
**Side**: Client-only (`runtime_side=client`)  
**Final result**: 72/72 builds green across Fabric, Forge, and NeoForge

## Challenges Encountered and Solutions

### Challenge 1: Forge 1.21 (51.x) — `AddGuiOverlayLayersEvent` not accessible

**Error**:
```
package net.minecraftforge.client.gui.overlay does not exist
import net.minecraftforge.client.event.AddGuiOverlayLayersEvent — cannot find symbol
```

**Investigation**: The auto source search found `AddGuiOverlayLayersEvent` and `ForgeLayeredDraw` in the decompiled Forge 51.x sources. However, the build still failed. This is NOT an AI source search error — the classes exist in the Forge source tree but are **not exported to the mod compile classpath** in Forge 51.0.33.

**Root cause**: Forge 51.x (1.21) introduced `ForgeLayeredDraw` and `AddGuiOverlayLayersEvent` internally but did not expose them as part of the public mod API. The old `RenderGuiOverlayEvent` and `VanillaGuiOverlay` were removed in 1.20.5, and the new system wasn't accessible until Forge 52.x (1.21.1).

**Solution**: Skip Forge 1.21 (51.x). Use `1.21.1` as the minimum Forge target for the 1.21 era.

---

### Challenge 2: Forge 1.21.6-1.21.8 (56-58.x) — Same accessibility issue

**Error**: Same as Challenge 1 — `AddGuiOverlayLayersEvent`, `ForgeLayer`, `ForgeLayeredDraw` not found.

**Investigation**: Auto source search for 1.21.6 found all these classes in the decompiled sources (6,790 Java files). This is **NOT an AI source search error** — the classes exist in the source but are not accessible from the mod classpath in Forge 56.x.

**Root cause**: Same as 51.x — the overlay API exists in Forge's internal code but is not part of the public mod API surface in these versions.

**Solution**: Skip Forge 1.21.6-1.21.8 (56-58.x). These versions have no accessible HUD rendering API for mods.

**Note**: This is a known Forge limitation for these specific versions. NeoForge equivalents (1.21.6-1.21.8) work fine via `RegisterGuiLayersEvent`.

---

### Challenge 3: Forge 1.21 — Wrong initial approach (RenderGuiOverlayEvent)

**Error**: Original code used `RenderGuiOverlayEvent` and `VanillaGuiOverlay` which were removed in Forge 1.20.5.

**Fix**: Switched to `AddGuiOverlayLayersEvent` + `ForgeLayeredDraw` pattern (same as 1.20.6).

---

### Challenge 4: Forge 1.21.6-1.21.8 — Wrong API classes

**Error**: Code used `GuiGraphicsExtractor` and `Identifier` which don't exist in 56.x.

**Investigation via AI Source Search**: Searched for `GuiGraphicsExtractor` in Forge 1.21.6 — found it. Searched for `Identifier` — found it. But the build still failed. This confirmed the accessibility issue (Challenge 2).

**Lesson**: The AI source search finding a class in decompiled sources does NOT guarantee it's accessible from the mod classpath. Always verify by actually building.

---

### Challenge 5: Forge 1.21.9-1.21.11 — `Identifier` not found

**Error**:
```
import net.minecraft.resources.Identifier — cannot find symbol
```

**Investigation**: Auto source search for 1.21.9 found 0 matches for `Identifier`. This IS a real API change — `Identifier` was renamed back to `ResourceLocation` in Forge 59.x.

**Fix**: Changed `FORGE_1219_PLUS_CLIENT` to use `ResourceLocation` instead of `Identifier`.

---

### Challenge 6: Forge 1.21.11 (61.x) — `ResourceLocation` not found

**Error**:
```
import net.minecraft.resources.ResourceLocation — cannot find symbol
```

**Investigation**: Auto source search for 1.21.11 found 0 matches for `ResourceLocation`. In Forge 61.x, `ResourceLocation` was renamed to `Identifier`.

**Fix**: Created separate `FORGE_1219_PLUS_CLIENT_61` that uses `Identifier` for 1.21.11 only. Split the 1.21.9-1.21.11 range into individual targets: 1.21.9, 1.21.10 (use `ResourceLocation`), 1.21.11 (uses `Identifier`).

---

### Challenge 7: Forge 1.21.6-1.21.8 — `GuiGraphicsExtractor` vs `GuiGraphics`

**Error**: Code used `GuiGraphicsExtractor` (correct for 56-58.x) but the build failed because the entire overlay package wasn't accessible.

**Investigation**: Source search confirmed `GuiGraphicsExtractor` exists in 56.x and `ForgeLayer.extract(GuiGraphicsExtractor, DeltaTracker)` is the correct interface. However, since the whole package is inaccessible, this is moot.

**Note for future**: When these versions become accessible, the correct API is:
- `ForgeLayer.extract(GuiGraphicsExtractor, DeltaTracker)` — NOT `render(GuiGraphics, DeltaTracker)`
- `Identifier` — NOT `ResourceLocation`
- `AddGuiOverlayLayersEvent.getBus(context.getModBusGroup()).addListener(...)` — EventBus 7 pattern

---

### Challenge 8: Forge 1.21.9+ — `ForgeLayer.render()` vs `ForgeLayer.extract()`

**Discovery**: In Forge 59.x (1.21.9+), `GuiGraphicsExtractor` was removed and `ForgeLayer` reverted to `render(GuiGraphics, DeltaTracker)`.

**Fix**: `FORGE_1219_PLUS_CLIENT` uses `GuiGraphics` and `ForgeLayer.render()` (via lambda `(gg, dt) -> { ... gg.drawString(...) }`).

### Challenge 9: Forge 26.1.2 — Duplicate `FORGE_261_CLIENT` definition

**Error**: The generator had TWO definitions of `FORGE_261_CLIENT`. The second (wrong) one used `GuiGraphics`/`LayeredDraw`/`ResourceLocation` and overwrote the first (correct) one that used `GuiGraphicsExtractor`/`ForgeLayer`/`Identifier`.

**Fix**: Removed the duplicate definition. The correct 26.1.2 API uses:
- `GuiGraphicsExtractor` (not `GuiGraphics`)
- `ForgeLayer.extract(GuiGraphicsExtractor, DeltaTracker)` (not `render`)
- `Identifier` (not `ResourceLocation`)
- `AddGuiOverlayLayersEvent.BUS.addListener(...)` (EventBus 7 static BUS)
- `gg.text(fr, text, x, y, color, shadow)` (not `drawString`)

### Challenge 15: Generator `get_failed_targets()` returning empty set

**Issue**: The `--failed-only` flag returned "No failed targets found" even though there were failures.

**Root cause**: The function first checks for `result.json` in the run directory. The top-level `result.json` has no `mods` array (it's the workflow-level result), so `result.get("mods", [])` returns empty. The function returned before checking the `artifacts/all-mod-builds/mods/` directory.

**Workaround**: Run the generator without `--failed-only` to regenerate all targets.

---

### Challenge 16: Multiple `FORGE_1216_PLUS_MOD` definitions not applying

**Issue**: Multiple `strReplace` operations failed because the text to replace had already been partially changed by a previous operation, making the exact match fail.

**Solution**: Read the exact current content of the file before each replacement, then use the exact text found.

## Key Lessons

### 1. AI Source Search finding a class ≠ class is accessible from mod classpath

The most important lesson from this port. In Forge 51.x and 56-58.x, `AddGuiOverlayLayersEvent` and `ForgeLayeredDraw` exist in the decompiled sources but are NOT part of the public mod API. The only way to know is to actually build.

### 2. ResourceLocation ↔ Identifier renaming happens across Forge versions

- Forge 59-60.x: `ResourceLocation` (standard Mojang name)
- Forge 61.x: `Identifier` (renamed)
- Forge 64.x (26.1.2): `Identifier` (still renamed)
- NeoForge 26.1.x: `Identifier`

Always use the AI Source Search to verify which name is correct for a specific version.

### 3. GuiGraphics ↔ GuiGraphicsExtractor split

- Forge 56-58.x: `GuiGraphicsExtractor` (new rendering system)
- Forge 59.x+: Back to `GuiGraphics` (reverted)
- NeoForge 26.1.x: `GuiGraphicsExtractor`
- Fabric 26.1.x: `GuiGraphics` (Mojang names)

### 4. ForgeLayer method name changed

- Forge 56-58.x: `ForgeLayer.extract(GuiGraphicsExtractor, DeltaTracker)`
- Forge 59.x+: `ForgeLayer.render(GuiGraphics, DeltaTracker)`

### 5. EventBus 7 registration patterns

- Forge 52-55.x: `context.getModEventBus().addListener(...)` (standard EventBus 6)
- Forge 56-58.x: `AddGuiOverlayLayersEvent.getBus(context.getModBusGroup()).addListener(...)` (EventBus 7 with `getBus`)
- Forge 59-60.x: `AddGuiOverlayLayersEvent.BUS.addListener(...)` (EventBus 7 static BUS, `getBus` deprecated)
- Forge 61.x+: `AddGuiOverlayLayersEvent.BUS.addListener(...)` (same)

### 6. GuiGraphicsExtractor.text() not drawString()

In versions using `GuiGraphicsExtractor`, the text rendering method is `text(Font, String, int, int, int, boolean)` — NOT `drawString()`. Using `drawString()` causes a compile error.

### 7. Fabric 26.1.x has no HUD rendering API

The Fabric API for 26.1.x only includes basic infrastructure (event system, lookups). `HudRenderCallback` was removed and no replacement exists yet. Skip these versions until the Fabric API catches up.

### 8. Split version ranges when API changes mid-range

The 1.21.9-1.21.11 range spans two different API eras (59-60.x uses `ResourceLocation`, 61.x uses `Identifier`). Always split ranges when the API changes within them.

### 9. Modrinth publish slug must match the project page URL exactly

The `--modrinth` flag and all `homepage=` fields in the generator script must use the exact Modrinth slug from the project's URL (`modrinth.com/mod/<slug>`). A wrong slug causes the entire publish step to fail with "project could not be found" — no versions get uploaded at all.

To verify the correct slug before running:
```
curl https://api.modrinth.com/v2/project/<slug-or-id> | python3 -m json.tool | grep slug
```
Or just open the project page in a browser and read the URL.

When fixing: search the generator script for ALL occurrences of the old slug — it typically appears in the docstring, `MOD_TXT_BASE`, and `fabric.mod.json` template functions. Then regenerate the bundle before re-running.

### Challenge 17: Modrinth publish — "project could not be found"

**Error**:
```
The Modrinth project could not be found.
```
`result.json` showed `project_input: "https://modrinth.com/mod/time-counter"` with `status: "failed"`.

**Investigation**:
The generator script (`scripts/generate_timecounter_bundle.py`) had the wrong Modrinth slug hardcoded as `time-counter` in three places:
- The docstring comment at the top
- `MOD_TXT_BASE` — the `homepage=` field written into every `mod.txt`
- `fabric_mod_json_presplit()` and `fabric_mod_json_split()` — the `"homepage"` field in `fabric.mod.json`

The slug was also passed as `--modrinth "https://modrinth.com/mod/time-counter"` when invoking `run_build.py`.

To find the correct slug, the Modrinth API was queried directly:
```
https://api.modrinth.com/v2/project/optimized-day-counter
```
This returned `"slug": "optimized-day-counter"` and `"id": "LtB4QBvY"`, confirming the correct URL is `https://modrinth.com/mod/optimized-day-counter`.

**Root cause**: The generator script was written with a placeholder slug (`time-counter`) that was never updated to match the actual Modrinth project slug (`optimized-day-counter`). The Modrinth project page URL (`/mod/optimized-day-counter`) is the canonical slug — not the internal project title or any other name.

**Fix**:
1. Replaced all three `time-counter` occurrences in `scripts/generate_timecounter_bundle.py` with `optimized-day-counter`
2. Regenerated the bundle (`python3 scripts/generate_timecounter_bundle.py`) so all `mod.txt` and `fabric.mod.json` files got the correct homepage URL
3. Re-ran `run_build.py` with `--modrinth "https://modrinth.com/mod/optimized-day-counter"`

**Result**: Publish succeeded. 7 new versions uploaded (Forge 1.21.9/10/11, Forge 26.1.2, NeoForge 26.1/26.1.1/26.1.2). All 65 previously-uploaded versions were correctly detected as already existing and skipped.

**Key lesson for future agents**: When the publish step fails with "project could not be found", the slug in the generator script is wrong. Verify the correct slug by:
1. Checking the Modrinth project page URL directly in a browser — the slug is the last path segment (e.g. `modrinth.com/mod/<slug>`)
2. Or querying `https://api.modrinth.com/v2/project/<slug-or-id>` and reading the `"slug"` field from the JSON response
3. Then update ALL occurrences in the generator script: the docstring, `MOD_TXT_BASE`, and any `fabric.mod.json` template functions
