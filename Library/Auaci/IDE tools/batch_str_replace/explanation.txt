# Batch String Replace Tool - Detailed Explanation

## Overview
The Batch String Replace tool performs multiple string replacements across one or more files in a single operation. It's more efficient than invoking str_replace multiple times and includes validation to ensure all replacements can be made before any are executed.

## Purpose
To enable efficient, coordinated multiple string replacements with built-in safety checks and flexible error handling options.

## Configuration
- **Tool Name**: `batch_str_replace`
- **Version**: 1.0.0
- **Execution Mode**: optional
- **Parameters**:
  - `replacements` (required, array): Array of replacement objects (1-50 items)
    - Each replacement object contains:
      - `path` (string): Path to the file to modify
      - `oldStr` (string): Exact string to find and replace (must match exactly one location)
      - `newStr` (string): New string to replace oldStr with
      - `description` (optional, string): Description of what this replacement does
  - `stopOnFirstError` (optional, boolean): If true, stop and rollback all replacements if any single replacement fails. Default: false
  - `atomic` (optional, boolean): If true, treat as atomic operation - all succeed or all fail. Default: false (allows partial success)

## Logic Flow
1. Validate all replacements before execution:
   - Check each file exists and is readable
   - Verify each `oldStr` matches exactly one location in its file
   - Ensure no more than 50 replacements total
2. If validation passes, execute replacements:
   - For each replacement, perform exact string match and replace (same logic as str_replace)
   - Apply changes to files
3. Error handling based on configuration:
   - `stopOnFirstError`: If any replacement fails, stop and rollback all successful replacements
   - `atomic`: All replacements must succeed or all are rolled back (note: full atomicity may depend on implementation)
   - Default (false): Continue with remaining replacements even if some fail, report results for each
4. Return detailed results for each replacement operation

## Key Features
- **Efficiency**: Single tool call for multiple replacements
- **Pre-validation**: All replacements validated before any execution
- **Flexible Error Handling**: Choose how to handle failures
- **Atomicity Options**: Support for all-or-nothing operations
- **Partial Success Reporting**: Detailed results for each replacement
- **Consistent Logic**: Uses same matching algorithm as str_replace for each replacement

## Error Handling
- **Retryable**: No (fail-fast on validation errors)
- **Common Errors**:
  - not found in file: One or more strings don't match any location → Use `view` to examine files
  - matches multiple: One or more strings match multiple locations → Add more context to make strings unique
  - too many replacements: Maximum 50 replacements per call → Split into multiple calls

## Usage Examples

### Coordinated API Changes
```json
{
  "replacements": [
    {
      "path": "/project/src/api/users.js",
      "oldStr": "app.get('/users/:id', getUserById);",
      "newStr": "app.get('/users/:id', authenticate, getUserById);",
      "description": "Add authentication middleware to user lookup"
    },
    {
      "path": "/project/src/api/posts.js",
      "oldStr": "app.get('/posts/:id', getPostById);",
      "newStr": "app.get('/posts/:id', authenticate, getPostById);",
      "description": "Add authentication middleware to post lookup"
    }
  ],
  "stopOnFirstError": false
}
```

### Bulk Variable Renaming
```json
{
  "replacements": [
    {
      "path": "/project/src/components/Header.js",
      "oldStr": "const [userData, setUserData] = useState(null);",
      "newStr": "const [userProfile, setUserProfile] = useState(null);",
      "description": "Rename state variable for clarity"
    },
    {
      "path": "/project/src/components/Header.js",
      "oldStr": "setUserData(response.data);",
      "newStr": "setUserProfile(response.data);",
      "description": "Update setter call to match new variable name"
    },
    {
      "path": "/project/src/components/Header.js",
      "oldStr": "return <div>{userData?.name}</div>;",
      "newStr": "return <div>{userProfile?.name}</div>;",
      "description": "Update JSX to use new variable name"
    }
  ],
  "atomic": true
}
```

## Best Practices
1. **Group Related Changes**: Use for replacements that are logically connected
2. **Consider Atomicity**: Set `atomic: true` when all changes must succeed together
3. **Use Descriptions**: Add descriptions to explain the purpose of each replacement
4. **Validate First**: Consider using individual `str_replace` calls first if unsure about matches
5. **Handle Failures**: Choose `stopOnFirstError` based on whether later changes depend on earlier ones
6. **Monitor Results**: Check individual replacement results to identify which succeeded/failed
7. **File Grouping**: Most efficient when replacements are in the same or few files

## Technical Implementation
The tool performs:
1. Pre-validation pass: Checks all replacements for validity
2. Execution pass: Applies each valid replacement using exact string matching
3. Error handling: Based on configuration flags
4. Result compilation: Detailed success/failure info for each replacement
5. Rollback capability: When configured, can revert successful changes on failure

## When to Use This Tool
- Making the same change across multiple similar files
- Coordinated changes that must happen together
- Bulk refactoring operations (renaming, API updates)
- Configuration updates across environments
- When you want to minimize tool call overhead

## Limitations
- Maximum 50 replacements per call
- All replacements validated upfront (may be slow for very large numbers)
- No support for cross-file dependencies in validation
- Atomicity implementation may vary (check specific implementation)

## Comparison with Similar Tools
- **vs str_replace**: Multiple replacements vs single replacement
- **vs apply_patch**: Line-oriented exact matches vs context-based fuzzy matching
- **vs edit_function**: Arbitrary string replacements vs semantic function replacements
- **vs edit_lines**: Content-based vs line-number-based operations
