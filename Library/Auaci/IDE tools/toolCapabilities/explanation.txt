# Tool Capabilities - Detailed Explanation

## Overview
The Tool Capabilities module provides utilities for detecting available tools, validating tool parameters, building execution contexts, and selecting the best tool for specific tasks. It's a foundational component that enables the IDE's tool system to work intelligently with different AI models and available tool sets.

## Purpose
To provide intelligent tool management capabilities including:
- Detecting which tools are available in the current environment
- Validating tool parameters against schemas
- Building execution contexts for AI models
- Selecting the optimal tool for common tasks (read, write, search, etc.)

## Configuration
This module exports several functions and constants:
- `ToolName`: Object containing standard tool name constants
- `detectToolCapabilities(availableTools)`: Function to detect available tools
- `validateToolParams(params, schema)`: Function to validate tool parameters
- `buildToolContext(availableTools, modelFamily)`: Function to build execution context
- `getBestToolForTask(context, taskType)`: Function to select best tool for task

## Core Components

### ToolName Constants
Standardized tool names used throughout the system:
- File operations: READ_FILE ('view'), CREATE_FILE ('create_file'), REPLACE_STRING ('str_replace'), etc.
- Search and discovery: FILE_SEARCH ('find_files'), GREP_SEARCH ('grep'), SEMANTIC_SEARCH ('semantic_search')
- Terminal/Shell: RUN_IN_TERMINAL ('run_command'), BASH ('bash')
- Web: WEB_SEARCH ('web_search'), WEB_FETCH ('web_fetch')
- Context and search: CONTEXT_SEARCH ('context_search')
- Code editing: APPLY_PATCH ('apply_patch'), EDIT_LINES ('edit_lines'), EDIT_FUNCTION ('edit_function')
- System: VIEW ('view'), LS ('ls'), TXTLIZE ('txtlize'), DIAGNOSTICS ('diagnostics')
- User interaction: ASK ('ask'), ATTEMPT_COMPLETION ('attempt_completion')
- Error recovery: ERROR_RECOVERER ('error_recoverer')
- Todo management: MANAGE_TODO_LIST ('manage_todo_list'), etc.
- File system: APPEND_FILE ('fs_append'), WRITE_FILE ('write_to_file')

## Logic Flow

### detectToolCapabilities(availableTools)
1. Initializes all tools as disabled (false)
2. If availableTools array provided, marks matching tools as enabled (true)
3. Computes derived capabilities:
   - hasEditTool: true if any of REPLACE_STRING, MULTI_REPLACE_STRING, or EDIT_FILE available
   - hasSearchTool: true if any of SEMANTIC_SEARCH, GREP_SEARCH, or FILE_SEARCH available
   - hasFileReadTool: true if READ_FILE ('view') available
4. Returns capabilities object with boolean flags

### validateToolParams(params, schema)
1. Returns empty error array if no schema or no properties
2. Checks required fields: adds error for each missing required parameter
3. Checks parameter types: validates each parameter against expected type in schema
4. Returns array of validation error messages (empty if valid)

### buildToolContext(availableTools, modelFamily)
1. Calls detectToolCapabilities to get capabilities
2. Stores availableTools array
3. Normalizes modelFamily to lowercase string
4. Builds toolsByName map for quick lookup by tool name
5. Returns context object with capabilities, availableTools, modelFamily, and toolsByName

### getBestToolForTask(context, taskType)
1. Extracts capabilities from context
2. Based on taskType, returns the best available tool:
   - 'read': Returns READ_FILE if hasFileReadTool is true
   - 'write': Returns REPLACE_STRING if available, else EDIT_FILE, else CREATE_FILE
   - 'multi-write': Returns MULTI_REPLACE_STRING if available
   - 'search': Returns SEMANTIC_SEARCH if available, else GREP_SEARCH, else FILE_SEARCH
   - 'terminal': Returns RUN_IN_TERMINAL if available
   - default: Returns null

## Integration with IDE
- Used by toolInvocationService for tool validation
- Used by index.js to build tool execution contexts
- Provides standardized tool names across the system
- Enables intelligent tool selection based on availability
- Supports model-specific tool hints through modelFamily parameter

## Best Practices
1. **Use ToolName Constants**: Always reference tools by ToolName constants to avoid typos
2. **Leverage Derived Capabilities**: Use hasEditTool, hasSearchTool, hasFileReadTool for quick checks
3. **Validate Early**: Use validateToolParams before executing tools to catch errors early
4. **Build Context Once**: Call buildToolContext once per conversation/task and reuse
5. **Let System Choose**: Use getBestToolForTask for common operations instead of hardcoding tool names
6. **Handle Missing Tools**: Always check if getBestToolForTask returns null before using

## Technical Implementation
The capabilities system:
1. Uses object reduction to initialize all tools as false
2. Uses Set for efficient lookup of available tool names
3. Computes derived capabilities through logical OR operations
4. Provides type-safe validation with clear error messages
5. Implements fallback logic for tool selection

## Usage Examples

### Checking Tool Availability
```javascript
const { detectToolCapabilities, ToolName } = require('./toolCapabilities');

const availableTools = [
  { name: 'view' },
  { name: 'str_replace' },
  { name: 'apply_patch' }
];

const capabilities = detectToolCapabilities(availableTools);

if (capabilities[ToolName.REPLACE_STRING]) {
  console.log('String replace tool is available');
}

if (capabilities.hasEditTool) {
  console.log('At least one edit tool is available');
}
```

### Validating Tool Parameters
```javascript
const { validateToolParams } = require('./toolCapabilities');

const schema = {
  type: 'object',
  properties: {
    path: { type: 'string' },
    oldStr: { type: 'string' },
    newStr: { type: 'string' }
  },
  required: ['path', 'oldStr', 'newStr']
};

const params = {
  path: '/src/file.js',
  oldStr: 'old text',
  newStr: 'new text'
};

const errors = validateToolParams(params, schema);
if (errors.length === 0) {
  // Parameters are valid
} else {
  console.error('Validation errors:', errors);
}
```

### Building Tool Context
```javascript
const { buildToolContext } = require('./toolCapabilities');

const availableTools = toolService.getTools(); // From toolInvocationService
const context = buildToolContext(availableTools, 'nemotron');

// Use context for tool selection
const bestReadTool = getBestToolForTask(context, 'read');
```

### Selecting Best Tool for Task
```javascript
const { getBestToolForTask, buildToolContext, ToolName } = require('./toolCapabilities');

const context = buildToolContext(availableTools);
const writeTool = getBestToolForTask(context, 'write');

if (writeTool === ToolName.REPLACE_STRING) {
  // Use str_replace for writing
} else if (writeTool === ToolName.EDIT_FILE) {
  // Use edit_file for writing
}
```

## Error Handling
- detectToolCapabilities: Gracefully handles non-array inputs
- validateToolParams: Returns clear error messages for missing/wrong types
- buildToolContext: Handles undefined availableTools and modelFamily
- getBestToolForTask: Returns null for unknown task types

## Benefits
1. **Standardization**: Consistent tool naming across the system
2. **Introspection**: Ability to query what tools are available
3. **Validation**: Prevents tool execution with invalid parameters
4. **Optimization**: Automatic selection of best available tool
5. **Model Adaptation**: Context can be tailored for different AI models
6. **Fallback Logic**: Graceful degradation when preferred tools unavailable

## Integration Points
- Used by toolInvocationService.validateToolInput()
- Used by index.js.getToolContext() and index.js.getToolCapabilities()
- Used by AI agent reasoning loop for tool selection
- Referenced in tool JSON documentation for consistency

## When to Use This Module
- Building tool execution contexts for AI models
- Validating tool parameters before execution
- Determining what capabilities are available in current environment
- Selecting appropriate tools for common operations
- Providing tool information to users or debugging systems
- Ensuring consistency in tool naming across the codebase

## Limitations
- Derived capabilities are based on simple OR logic (may need refinement for complex scenarios)
- Tool selection logic is simplistic (always returns first match in priority list)
- Does not consider tool performance or cost characteristics
- Validation is basic type checking (does not support complex JSON Schema features)
