# Apply Patch Tool - Detailed Explanation

## Overview
The Apply Patch tool is a powerful code modification utility that uses a 6-pass fuzzy matching engine to apply unified patches to files. It supports creating, updating, and deleting files using a standardized patch format similar to git diff but with enhanced matching capabilities.

## Purpose
To enable precise, context-aware modifications to files through unified patch syntax, allowing for complex multi-file edits with automatic indentation handling and fuzzy matching for imperfect AI-generated patches.

## Configuration
- **Tool Name**: `apply_patch`
- **Version**: 2.0.0
- **Execution Mode**: optional
- **Parameters**:
  - `input` (required, string): The full patch text starting with '*** Begin Patch' and ending with '*** End Patch'
  - `explanation` (optional, string): Short intent summary used for healing or debugging
  - `diff` (deprecated, string): Alias for input parameter

## Patch Format
```
*** Begin Patch
*** Update File: /absolute/path/to/file.ext
@@ optional_context_hint @@
 context line (unchanged)
-removed line
+added line
 context line (unchanged)
*** End Patch
```

## Supported Operations
1. **Update File**: Edit an existing file using context-based hunks
2. **Add File**: Create a new file (all lines prefixed with `+`)
3. **Delete File**: Remove an existing file
4. **Move to**: Rename/move a file (used after Update File block)

## Hunk Rules
- Each hunk starts with `@@` (optionally followed by context hint)
- Lines starting with space are context (must exist in file)
- Lines starting with `-` are deleted
- Lines starting with `+` are inserted
- Include 1-3 lines of context before/after changes for accurate matching
- Multiple hunks per file separated by `@@`
- `*** End of File` marks final hunk at file end

## Matching Engine (6-Pass Fuzzy Matching)
1. **Exact Match**: Try to find exact context match
2. **Whitespace Tolerant**: Ignore whitespace differences in context
3. **Line-Ending Tolerant**: Handle different line endings (LF/CRLF)
4. **Indentation Tolerant**: Convert tabs↔spaces automatically
5. **Fuzzy Match**: Use Levenshtein distance for approximate matching
6. **Loose Fallback**: Last resort matcher for imperfect patches

## Error Handling
- **Retryable**: Yes (with exponential backoff)
- **Common Errors**:
  - Invalid context: Suggest using `view` tool to re-examine file
  - Missing file: Suggest using `view` to confirm file exists
  - Missing End Patch: Ensure patch ends with '*** End Patch'
  - Duplicate file paths: Combine edits under single Update File block

## Usage Examples

### Simple String Replacement
```
*** Begin Patch
*** Update File: /project/src/main.js
@@
 function hello() {
-  console.log("Hello World");
+  console.log("Hello Auaci");
 }
*** End Patch
```

### Adding a New File
```
*** Begin Patch
*** Add File: /project/src/new-feature.js
+function newFeature() {
+  return "This is a new feature";
+}
*** End Patch
```

### Deleting a File
```
*** Begin Patch
*** Delete File: /project/src/obsolete.js
*** End Patch
```

### Multi-File Patch
```
*** Begin Patch
*** Update File: /project/src/file1.js
@@
- oldCode1
+ newCode1
*** Update File: /project/src/file2.js
@@
- oldCode2
+ newCode2
*** End Patch
```

## Integration with IDE
- Loaded automatically by ToolInvocationService
- Validated against JSON schema before execution
- Supports error recovery through Error Recoverer tool
- Results formatted for AI model consumption in `<tool_result>` tags
- Respects excluded folders configuration for security

## Best Practices
1. Use absolute paths when possible
2. Provide sufficient context (2-3 lines) around changes
3. Ensure context lines match actual file content exactly
4. For large files, include more context to avoid ambiguity
5. Use explanation parameter to document intent for debugging
6. Validate patch syntax before submission
7. Combine related changes in single patch when possible

## Technical Implementation
The tool uses a sophisticated matching algorithm that:
1. Normalizes line endings and indentation
2. Applies multiple matching strategies in order of precision
3. Falls back to fuzzy matching when exact matches fail
4. Handles moved lines and blocks intelligently
5. Provides detailed error messages for debugging
